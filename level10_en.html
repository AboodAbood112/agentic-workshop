1 The Problem: You are the Orchestrator

In Levels 1-9 you built Agents, Workflows, and Triggers — but you were deciding which Agent to use, what it should do, and when. This doesn't scale.

⚠️ Common Mistake

Making the Orchestrator too smart. Its role is delegation, not execution. If it writes code itself, you've built a giant Agent, not an Orchestrator.

2 CRUD on Agents

🟢 Create
Spawn an Agent with specific goals and tools
🔵 Read
Check status, retrieve outputs, inspect logs
🟡 Update
Send new commands, modify instructions
🔴 Delete
Stop the Agent when its job is done

Agent Lifecycle:

Create → Command → Monitor → (Command if needed) → Read results → Delete

Cleanup Policy: Delete One-Shot agents after completion. Keep Multi-phase agents for inspection then delete them.

3 Orchestrator Patterns

⚡ Pattern 1: One-Shot Agent

For clear, simple tasks — One Agent, One Task

Analyze → Create agent → Command → Monitor → Collect results → Delete → Report

Example: "Fix this typo", "Update this dependency"

🔭 Pattern 2: Scout and Build

Two phases — Scout first, then execute

Create scout (read-only, fast model)
Create builder (full tools, capable model)
Run scout → wait → extract findings
Pass findings to builder → Run builder
Retain both → Report

Example: "Add a priority field to TaskFlow"

⚙️ Pattern 3: ADW Dispatch

Runs a full ADW from Level 9

Analyze → Choose: one-shot / scout-build / ADW?
→ Start ADW (plan_build, plan_build_review...)
→ Monitor → Report

4 Context Flow Between Agents

ORCHESTRATOR — The Central Memory

1. User request: "Add priority field"
↓
2. Scout phase
└─> Scout receives: User request
<── Scout returns: List of files + analysis
↓
3. Build phase
└─> Builder receives: User request + Scout results
<── Builder returns: Modified files + summary
↓
4. Report: Consolidates phase results

⚠️ Common Mistake

Passing more context than necessary to Sub-agents. The Orchestrator must summarize, not stack. 500 lines of Scout analysis → the Builder only needs the key findings.

5 Practical Exercise Steps

Create Orchestrator Skill

Create .pi/skills/orchestrator/SKILL.md — Its role (Lead Engineer/Delegator), the patterns it uses, when to use each pattern, its lifecycle policy. No file editing tools — it delegates, it doesn't execute.

2
Create One-Shot Command

Create orch_one_shot_agent.md — Accepts the task as $ARGUMENTS, creates appropriate Agent, waits for completion, collects results, deletes Agent, reports.

3
Create Scout-and-Build Command

Create orch_scout_and_build.md — Scout (read-only + fast model) then Builder (full tools + strong model + Scout results). The Orchestrator summarizes Scout results and passes them on.

4
Test with Tasks of Varying Complexity
Simple:  "Fix the typo in the TaskFlow README"
Medium:  "Add a task priority field to TaskFlow"
Complex: "Refactor the TaskFlow API error handling"

Does the Orchestrator pick the right pattern for each task?

6 Key Takeaways

🎼The Orchestrator delegates, doesn't execute — decides what to do, spawns Agents to do it, monitors the results.
🗄️CRUD is the mental model — Create, Read, Update, Delete with an explicit cleanup policy for each pattern.
🧵Context flows through the Orchestrator — The results of each phase become inputs to the next — the Orchestrator is the memory.
🧠Pattern selection is the Orchestrator's value — One-shot for simple, Scout-and-Build for complex, ADW for multi-step.
🧹Explicit Cleanup Policy — Delete One-shots quickly, retain Multi-phase for inspection.
