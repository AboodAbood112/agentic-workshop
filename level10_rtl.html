٢ CRUD على الـ Agents

🟢 Create
أنشئ Agent بغرض وأدوات محددة
🔵 Read
تحقق من الحالة، استرد المخرجات، افحص السجلات
🟡 Update
أرسل أوامر جديدة، عدّل التعليمات
🔴 Delete
أوقف الـ Agent عند انتهاء عمله

دورة حياة الـ Agent:

Create → Command → Monitor → (Command if needed) → Read results → Delete

سياسة Cleanup: احذف One-Shot agents بعد الانتهاء. احتفظ بـ Multi-phase agents للفحص ثم احذفهم.

٣ أنماط الـ Orchestrator

⚡ Pattern 1: One-Shot Agent

للمهام الواضحة البسيطة — Agent واحد، مهمة واحدة

Analyze → Create agent → Command → Monitor → Collect results → Delete → Report

مثال: "أصلح هذا الخطأ الإملائي"، "حدّث هذه التبعية"

🔭 Pattern 2: Scout and Build

مرحلتان — استطلع أولاً ثم نفّذ

Create scout (read-only, fast model)
Create builder (full tools, capable model)
Run scout → wait → extract findings
Pass findings to builder → Run builder
Retain both → Report

مثال: "أضف حقل priority لـ TaskFlow"

⚙️ Pattern 3: ADW Dispatch

يُشغّل ADW كاملاً من Level 9

Analyze → Choose: one-shot / scout-build / ADW?
→ Start ADW (plan_build, plan_build_review...)
→ Monitor → Report

٤ تدفق السياق بين الـ Agents

ORCHESTRATOR — الذاكرة المركزية

1. طلب المستخدم: "Add priority field"
↓
2. Scout phase
└─> Scout يستقبل: طلب المستخدم
<── Scout يُعيد: قائمة ملفات + تحليل
↓
3. Build phase
└─> Builder يستقبل: طلب المستخدم + نتائج Scout
<── Builder يُعيد: ملفات تغيّرت + ملخص
↓
4. Report: يجمع نتائج المرحلتين

⚠️ خطأ شائع

تمرير سياق أكثر مما يلزم للـ Sub-agents. الـ Orchestrator يجب أن يُلخّص لا يُكدّس. 500 سطر من تحليل Scout → الـ Builder يحتاج الـ key findings فقط.

٥ خطوات التمرين العملي

١
أنشئ Orchestrator Skill

أنشئ .pi/skills/orchestrator/SKILL.md — دوره (مهندس رئيسي/مفوّض)، الأنماط التي يستخدمها، متى يستخدم كل نمط، سياسة الـ lifecycle. بدون أدوات تعديل ملفات — يُفوّض لا ينفّذ.

٢
أنشئ One-Shot Command

أنشئ orch_one_shot_agent.md — يقبل الـ task كـ $ARGUMENTS، ينشئ Agent مناسب، ينتظر الانتهاء، يجمع النتائج، يحذف الـ Agent، يُبلّغ.

٣
أنشئ Scout-and-Build Command

أنشئ orch_scout_and_build.md — Scout (read-only + نموذج سريع) ثم Builder (أدوات كاملة + نموذج قوي + نتائج Scout). الـ Orchestrator يُلخّص نتائج Scout ويمررها.

٤
اختبر بمهام متفاوتة التعقيد
بسيط:  "Fix the typo in the TaskFlow README"
متوسط: "Add a task priority field to TaskFlow"
معقد:  "Refactor the TaskFlow API error handling"

هل الـ Orchestrator يختار النمط المناسب لكل مهمة؟

٦ الدروس المستفادة

🎼الـ Orchestrator مفوّض لا منفّذ — يقرر ما يُفعل، يُنشئ Agents لتفعله، يراقب النتائج.
🗄️CRUD هو النموذج الذهني — Create، Read، Update، Delete مع سياسة cleanup صريحة لكل نمط.
🧵السياق يتدفق عبر الـ Orchestrator — نتائج كل مرحلة تصبح مدخلات المرحلة التالية — الـ Orchestrator هو الذاكرة.
🧠اختيار النمط هو قيمة الـ Orchestrator — One-shot للبسيط، Scout-and-Build للمعقد، ADW للمتعدد الخطوات.
🧹Cleanup policy صريحة — احذف One-shot سريعاً، احتفظ بـ Multi-phase للفحص.
