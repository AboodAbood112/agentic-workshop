🎻 Level 11: Orchestrator Developer Workflows

Class 3, Gear 2 — The Orchestrator Drives a Full Development Workflow

"The Orchestrator doesn't just manage Agents — it leads a complete development workflow, creating specialized teams for different problems."

1 From a Single Agent to a Complete Workflow

Level 10 — One Agent at a time:

Orchestrator → create("builder") → command("build X") → done

Level 11 — Integrated Workflow:

Orchestrator → Scout Phase (Parallel exploration)
            → Plan Phase  (With Scout results)
            → Build Phase (With approved plan)
            → Review Phase (Quality check)
            → Fix Phase   (If review fails)
            → Report      (Comprehensive final report)
💡 Difference: A manager delegating tasks vs. an orchestra conductor ensuring harmony among instruments

2 Designing Specialized Worker Agents

Principle: Fewest possible tools for each job — extra tools = temptation to stray from the task

🔍 Scout — Exploration
Tools: Read, Grep, Glob only
Model: Fast and lightweight (Cheap)
Output: Relevant files + patterns + suggestion
❌ Never any write tools
📐 Planner — Planning
Tools: Read, Write (specs only), Shell
Model: Powerful (Deep thinking)
Output: Structured execution plan in specs/
❌ Does not execute — plans only
🔨 Builder — Execution
Tools: Read, Write, Edit, Shell, Tests
Model: Most capable (Most expensive)
Output: Modified files + build report
⚡ Executes the plan precisely — no improvisation
🔎 Reviewer — Review
Tools: Read, Shell (tests/lint)
Model: Powerful (Deep code understanding)
Output: PASS/FAIL + structured results
❌ No write tools — independent reviewer

3 The Pentagonal Workflow — 5 Phases

Phase 1
Scout
3 Parallel Scouts: src/ + tests/ + configs/
Merge results into a single report
🤔 Decision: Are Scout results sufficient? Yes → Continue
Phase 2
Plan
Input: User request + Scout report
Output: Execution plan in specs/
🤔 Decision: Is the plan complete? Yes → Continue
Phase 3
Build
Input: Plan only (No raw Scout data)
Output: Modified files + passing tests
🤔 Decision: Did the build succeed? Yes → Continue
Phase 4
Review
Input: Changes and diff only
Output: PASS ✅ or FAIL ❌ + results
🤔 Decision: PASS or FAIL?
Phase 4.5
Fix (Conditional)
Only if FAIL — fixes then re-reviews
Phase 5
Report
Gathers: Scout + Plan + Build + Review + Fix
→ Comprehensive final report to the user

4 Context Rule: Each Agent Sees a Curated Slice

Scout
Receives:
User request + Target folder only

Planner
Receives:
User request + Scout results (summarized)

Builder
Receives:
Plan only — no Scout, no Planner thoughts

Reviewer
Receives:
Changes and diff only — no plan, no context

Orchestrator
Sees:
Everything — it is the central memory

⚠️ Common Mistake

Passing all context to every Agent. The Orchestrator summarizes and curates — it doesn't pile on. A 500-line Scout → Builder needs only the key findings.

6 Key Takeaways

🎼Driven Workflows > Managed Agents — 5 phases with decision points are far more powerful than individual Agents.
🎯Minimal Capabilities per Agent — Scout reads. Planner plans. Builder executes. Reviewer evaluates. Every extra tool = temptation to deviate.
✂️Context Curation is the Orchestrator's Job — Don't overload every Agent. Give each only what it needs.
🔀Decision Points Make the Workflow Adaptive — Without them, you have a shell script. With them, you have an intelligent system.
🚫The Orchestrator Doesn't Play — It doesn't write code, modify files, or run tests. It only delegates.
