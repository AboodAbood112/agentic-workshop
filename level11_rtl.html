🎻 Level 11: Orchestrator Developer Workflows

Class 3, Gear 2 — الـ Orchestrator يقود سير عمل تطوير كامل

"الـ Orchestrator لا يُدير Agents فقط — يقود سير عمل تطوير كاملة، يُنشئ فرقاً متخصصة لمشاكل مختلفة."

١ من Agent واحد إلى سير عمل كاملة

Level 10 — Agent واحد في كل مرة:

Orchestrator → create("builder") → command("build X") → done

Level 11 — سير عمل متكاملة:

Orchestrator → Scout Phase (استطلاع موازي)
            → Plan Phase  (مع نتائج Scout)
            → Build Phase (مع الخطة المعتمدة)
            → Review Phase (تحقق من الجودة)
            → Fix Phase   (إذا فشلت المراجعة)
            → Report      (تقرير نهائي شامل)
💡 الفرق: مدير يُفوّض مهام vs. قائد أوركسترا يضمن التناسق بين الآلات

٢ تصميم الـ Worker Agents المتخصصين

مبدأ: أقل أدوات ممكنة لكل وظيفة — الأدوات الزائدة = إغراء للخروج عن المهمة

🔍 Scout — الاستطلاع
الأدوات: Read، Grep، Glob فقط
النموذج: سريع وخفيف (رخيص)
المخرج: ملفات ذات صلة + أنماط + اقتراح
❌ بدون أدوات كتابة أبداً
📐 Planner — التخطيط
الأدوات: Read، Write (specs فقط)، Shell
النموذج: قوي (تفكير عميق)
المخرج: خطة تنفيذ منظمة في specs/
❌ لا ينفّذ — يخطط فقط
🔨 Builder — التنفيذ
الأدوات: Read، Write، Edit، Shell، Tests
النموذج: الأكثر قدرة (أغلى)
المخرج: ملفات معدّلة + تقرير بناء
⚡ ينفّذ الخطة بدقة — لا ارتجال
🔎 Reviewer — المراجعة
الأدوات: Read، Shell (tests/lint)
النموذج: قوي (فهم عميق للكود)
المخرج: PASS/FAIL + نتائج منظمة
❌ بدون أدوات كتابة — مراجع مستقل

٣ سير العمل الخماسي — 5 مراحل

Phase 1
Scout
3 Scouts بالتوازي: src/ + tests/ + configs/
تدمج النتائج في تقرير واحد
🤔 قرار: هل نتائج Scout كافية؟ نعم → تابع
Phase 2
Plan
المدخل: طلب المستخدم + تقرير Scout
المخرج: خطة تنفيذ في specs/
🤔 قرار: هل الخطة مكتملة؟ نعم → تابع
Phase 3
Build
المدخل: الخطة فقط (لا بيانات Scout الخام)
المخرج: ملفات معدّلة + اختبارات تمر
🤔 قرار: هل البناء نجح؟ نعم → تابع
Phase 4
Review
المدخل: التغييرات والـ diff فقط
المخرج: PASS ✅ أو FAIL ❌ + نتائج
🤔 قرار: PASS أو FAIL؟
Phase 4.5
Fix (شرطي)
فقط إذا FAIL — يُصلح ثم يُعيد المراجعة
Phase 5
Report
يجمع: Scout + Plan + Build + Review + Fix
→ تقرير نهائي شامل للمستخدم

٤ قاعدة السياق: كل Agent يرى شريحة مُنتقاة

Scout
يستقبل:
طلب المستخدم + المجلد الهدف فقط

Planner
يستقبل:
طلب المستخدم + نتائج Scout (ملخصة)

Builder
يستقبل:
الخطة فقط — لا Scout، لا تفكير Planner

Reviewer
يستقبل:
التغييرات والـ diff فقط — لا خطة، لا سياق

Orchestrator
يرى:
كل شيء — هو الذاكرة المركزية

⚠️ خطأ شائع

تمرير كل السياق لكل Agent. الـ Orchestrator يُلخّص ويُنتقي — لا يُكدّس. Scout بـ 500 سطر → Builder يحتاج الـ key findings فقط.

٥ خطوات التمرين العملي

١
أنشئ Scout Skill

أنشئ .pi/skills/scout-report-suggest/SKILL.md — قراءة فقط، نموذج خفيف، 3 أدوات read-oriented. المخرج: ملفات ذات صلة + أنماط + اقتراح مقاربة.

٢
أنشئ Planner Skill

أنشئ .pi/skills/planner/SKILL.md — نموذج قوي للتفكير العميق. يقرأ الـ codebase، يكتب الخطة في specs/. لا تنفيذ — تخطيط فقط.

٣
أنشئ Builder Skill

أنشئ .pi/skills/build-agent/SKILL.md — الأدوات الكاملة + أقوى نموذج. ينفّذ الخطة بدقة، يُشغّل الاختبارات، يُبلّغ بما تغيّر.

٤
أنشئ 5-Phase Workflow Command

أنشئ orch_plan_w_scouts_build_review.md — 5 مراحل صريحة مع نقاط قرار بين كل مرحلتين. Phase 1 ثلاث Scouts بالتوازي.

٥
أنشئ Multi-Team Command

أنشئ orch_multi_team.md — فرق متوازية لـ streams مختلفة (Frontend + Backend). تنسيق العقود المشتركة (API interface) بين الفرق.

٦ الدروس المستفادة

🎼Workflows مُقادة > Agents مُدارة — 5 مراحل بنقاط قرار أقوى بكثير من Agents فردية.
🎯قدرات أدنى لكل Agent — Scout يقرأ. Planner يُخطط. Builder ينفّذ. Reviewer يُقيّم. كل أداة زائدة = إغراء للخروج عن المهمة.
✂️انتقاء السياق مسؤولية الـ Orchestrator — لا تُكدّس على كل Agent. أعطِ كلاً ما يحتاجه فقط.
🔀نقاط القرار تجعل الـ Workflow تكيفية — بدونها عندك shell script. بها عندك نظام ذكي.
🚫الـ Orchestrator لا يعزف — لا يكتب كوداً، لا يعدّل ملفات، لا يُشغّل اختبارات. يُفوّض فقط.
