🔧 Level 3: Skills, MCPs & Custom Tools

Class 1, Gear 3 — Expanding the Agent's Capabilities

"Sub-agents, Skills, and MCPs all give you the same thing — custom tools. The difference is only in the packaging."

1 What are Skills?

Reusable capabilities defined in .pi/skills/ — each skill is in its own folder containing instructions + templates + examples.

📁 File Structure:

.pi/skills/
└── skill-name/
    ├── SKILL.md          # Skill definition
    └── templates/
        └── template.md  # Ready-made templates

🆚 Difference between Sub-Agent and Skill:

FeatureSub-AgentSkill
Location.pi/prompts/agents/.pi/skills/
StructureSingle fileFolder with templates
InvocationAuto-delegation/ command or delegation
ContextIndependent contextWorks within current context
ModelSpecifies its ownUses session model
💡 Rule of Thumb: If the task is independent → Sub-Agent. If it needs templates or works within current context → Skill.

⚠️ Common Mistake

Creating a Skill when a Sub-Agent is enough. Don't create a Skill unless you have real content to put in the folder.

2 What are MCP Servers?

Model Context Protocol — external processes that provide tools beyond the core four tools.

🛠️ Examples of MCPs:

MCPCapability
FirecrawlConvert any URL into structured content
PlaywrightControl a real browser for testing
Database MCPDirect querying of databases
Slack / GitHub / JiraIntegration with workflow tools

⚙️ Configuration file .mcp.json:

{
  "mcpServers": {
    "firecrawl": {
      "command": "npx",
      "args": ["-y", "firecrawl-mcp"],
      "env": { "FIRECRAWL_API_KEY": "your-key" }
    },
    "playwright": {
      "command": "npx",
      "args": ["-y", "@anthropic/mcp-playwright"],
      "env": {}
    }
  }
}

⚠️ Common Mistake

Saving API keys in the repo! Put .mcp.json in .gitignore and share .mcp.json.sample without real keys.

3 Custom Tools via Prime Prompts

Sometimes you don't need a Skill or MCP — it's enough to describe the pattern in AGENTS.md:

## Custom Tool: Research Pattern
When you need to research a library:
1. Search the web for the official docs
2. Fetch the getting-started page
3. Save a summary to ai_docs/<library-name>.md
4. Report what you learned

🔒 Permissions .pi/settings.json:

{
  "permissions": {
    "allow": [
      "Bash(bun run:*)",
      "Bash(git status)",
      "Bash(git diff:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push:*)",
      "Bash(git reset --hard:*)"
    ]
  }
}

4 Practical Exercise Steps

1
Create meta-agent skill

Create .pi/skills/meta-agent/SKILL.md — a Skill version of meta-agent with reference to a template.

2
Create the Agent Template

Create .pi/skills/meta-agent/templates/subagent-template.md with placeholders like <AGENT_NAME>, <TOOL_LIST>, <ROLE>.

3
Create MCP Config Sample

Create .mcp.json.sample with Firecrawl and Playwright — use empty strings for API keys.

4
Create Permissions File

Create .pi/settings.json — allow safe dev commands, and block destructive ones.

🔍 Checking your version:

ls .pi/skills/meta-agent/           # SKILL.md  templates/
ls .pi/skills/meta-agent/templates/ # subagent-template.md
cat .mcp.json.sample | python3 -m json.tool
cat .pi/settings.json | python3 -m json.tool

5 Key Takeaways

📐Skills = Templated Capabilities — guarantees the same structure and quality every time.
🌐MCPs = External Tools — connects the Agent to APIs, browsers, and databases.
📝Custom Tools = Smart Instructions — describe the pattern in the prime prompt and the Agent follows it without setup.
🔒settings.json = Safety Net — controls what runs automatically and what requires your approval.
📈Start Simple — Custom tool first, then Skill when you need templates, then MCP when you need external access.
💡 All three serve the same goal
Skills + MCPs + Custom Tools = Expanding the Core Four (Read, Write, Edit, Bash) 🚀
