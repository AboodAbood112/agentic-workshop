🔁 Level 4: Closed-Loop Prompts

Class 1, Gear 4 — Agents verifying their own work

"An Agent building without verification is like a developer who never runs their tests. More computation on verification = higher confidence in results."

2 The Three Commands

CommandRolePattern
/buildExecute the planRead plan → Code → Run tests → Report
/reviewAnalyze the workRead diffs → Classify issues → Final verdict
/fixFix issuesRead report → Fix by priority → Verify
/build → /review → /fix → /review → ... PASS ✅

3 Details of each command

🔨 /build — Execution

  • Read the plan file to understand what to build
  • Understand the current state — read files and check AGENTS.md
  • Execute changes in the order specified in the plan
  • Run tests and type checks — mandatory
  • Fix errors and re-verify until everything passes
  • Generate a build report (PASS/FAIL + changes + verification results)

🔍 /review — Review (4 Severity Levels)

🚨 BLOCKER — Must fix → Verdict: FAIL

Security vulnerabilities, API breakage, risk of data loss, exposed credentials

🔴 HIGH RISK — Must fix

Performance regressions, missing error handling, memory leaks, missing features

🟡 MEDIUM RISK — Fix soon

Code duplication, inconsistent naming, missing tests

🟢 LOW RISK — Nice to have

Style differences, aesthetic improvements

📁 Saves the report in: app_review/review_<timestamp>.md | Uses model: opus for deeper analysis

🔧 /fix — The Fix

  • Extract issues from the review report, sorted by priority
  • Fix Blockers first, then High, then Medium, then Low
  • Verify each fix individually
  • Run full verification after all fixes
  • Generate a fix report in app_fix_reports/fix_<timestamp>.md

4 Why More Compute = Higher Confidence?

ScenarioCostResult
Build only (open loop)LowQuality not guaranteed, you have to review
Build + Review + FixHigher × 4Reviewed code, documented report, high confidence
💡 A review that catches a security vulnerability costs cents. The same vulnerability in production costs a fortune.

5 Practical Exercise Steps

1
Create /build command

Create .pi/prompts/build.md — accepts the plan file path via $ARGUMENTS, executes, and forcefully verifies.

2
Create /review command

Create .pi/prompts/review.md — analyzes git diffs, categorizes issues by 4 levels, ends with PASS/FAIL.

3
Create /fix command

Create .pi/prompts/fix.md — reads the review report, fixes by priority, documents before/after.

4
Test the full loop

Write a small plan ← /build ← /review ← if FAIL: /fix ← /review again until PASS.

5
Inspect the reports

Read the files in app_review/ and app_fix_reports/ — are they organized and helpful?

🔍 Checking your version:

ls .pi/prompts/                    # build.md  review.md  fix.md
grep "ARGUMENTS" .pi/prompts/build.md
grep "app_review" .pi/prompts/review.md
grep "app_fix_reports" .pi/prompts/fix.md

6 Key Takeaways

🔁Closed loops catch bugs — before they reach you, the Agent reviews its own work.
🎯Severity levels determine priorities — Blockers first always, don't waste compute on style before safety.
📋Reports create accountability — documented files show what was built, what was reviewed, and what was fixed.
💰More compute = higher confidence — verification tokens are much cheaper than production bugs.
🔗This is the culmination — AGENTS.md + Sub-Agents + AI Docs + Skills + Closed Loop = a complete agentic system.
