🔁 Level 4: Closed-Loop Prompts

Class 1, Gear 4 — Agents تتحقق من عملها بنفسها

"Agent يبني بدون تحقق، مثل مطوّر لا يشغّل اختباراته أبداً. المزيد من الحوسبة على التحقق = ثقة أكبر في النتائج."

١ الفرق بين Open Loop و Closed Loop

❌ Open Loop — "أتمنى أن يكون صح"

Request
↓
Implement
↓
Done 🤞

✅ Closed Loop — "تأكدت أن الكود صح"

Request
↓
Implement
↓
Validate
↓
Fix
↓
Validate ✅

٢ الأوامر الثلاثة

الأمرالدورالنمط
/buildتنفيذ الخطةاقرأ الخطة → كوّد → شغّل الاختبارات → أبلّغ
/reviewتحليل العملاقرأ الـ diffs → صنّف المشاكل → حكم نهائي
/fixإصلاح المشاكلاقرأ التقرير → أصلح بالأولوية → تحقق
/build → /review → /fix → /review → ... PASS ✅

٣ تفاصيل كل أمر

🔨 /build — التنفيذ

  • اقرأ ملف الخطة لفهم ما يجب بناؤه
  • افهم الحالة الراهنة — اقرأ الملفات وتحقق من AGENTS.md
  • نفّذ التغييرات بالترتيب المحدد في الخطة
  • شغّل الاختبارات وفحص الأنواع — لا خيار
  • أصلح الأخطاء وأعد التحقق حتى ينجح كل شيء
  • أنتج تقرير بنية (PASS/FAIL + التغييرات + نتائج التحقق)

🔍 /review — المراجعة (4 مستويات خطورة)

🚨 BLOCKER — يجب الإصلاح → الحكم: FAIL

ثغرات أمنية، كسر الـ API، خطر فقدان البيانات، credentials مكشوفة

🔴 HIGH RISK — يجب الإصلاح

تراجع الأداء، غياب معالجة الأخطاء، تسرب الذاكرة، ميزات ناقصة

🟡 MEDIUM RISK — أصلح قريباً

تكرار الكود، تسمية غير متسقة، اختبارات ناقصة

🟢 LOW RISK — حسن لو أُصلح

اختلافات الأسلوب، تحسينات جمالية

📁 يحفظ التقرير في: app_review/review_<timestamp>.md | يستخدم model: opus لتحليل أعمق

🔧 /fix — الإصلاح

  • استخرج المشاكل من تقرير المراجعة مرتبة بالأولوية
  • أصلح Blockers أولاً، ثم High، ثم Medium، ثم Low
  • تحقق من كل إصلاح على حدة
  • شغّل التحقق الكامل بعد كل الإصلاحات
  • أنتج تقرير إصلاح في app_fix_reports/fix_<timestamp>.md

٤ لماذا الحوسبة الأكثر = ثقة أكبر؟

السيناريوالتكلفةالنتيجة
Build فقط (open loop)منخفضةجودة غير مضمونة، أنت تراجع
Build + Review + Fixأعلى × 4كود مراجع، تقرير موثق، ثقة عالية
💡 مراجعة تكتشف ثغرة أمنية تكلف سنتات. نفس الثغرة في الـ production تكلف ثروة.

٥ خطوات التمرين العملي

١
أنشئ /build command

أنشئ .pi/prompts/build.md — يقبل مسار ملف الخطة عبر $ARGUMENTS، يُنفّذ، ويتحقق إلزامياً.

٢
أنشئ /review command

أنشئ .pi/prompts/review.md — يحلل git diffs، يصنّف المشاكل بـ 4 مستويات، ينتهي بـ PASS/FAIL.

٣
أنشئ /fix command

أنشئ .pi/prompts/fix.md — يقرأ تقرير المراجعة، يصلح بالأولوية، يوثق قبل/بعد.

٤
اختبر الحلقة الكاملة

اكتب خطة صغيرة ← /build ← /review ← إذا FAIL: /fix ← /review مجدداً حتى PASS.

٥
افحص التقارير

اقرأ الملفات في app_review/ وapp_fix_reports/ — هل هي منظمة ومفيدة؟

🔍 التحقق:

ls .pi/prompts/                    # build.md  review.md  fix.md
grep "ARGUMENTS" .pi/prompts/build.md
grep "app_review" .pi/prompts/review.md
grep "app_fix_reports" .pi/prompts/fix.md

٦ الدروس المستفادة

🔁الحلقات المغلقة تمسك الأخطاء — قبل أن تصلك، الـ Agent يراجع عمله بنفسه.
🎯مستويات الخطورة تُحدد الأولويات — Blockers أولاً دائماً، لا تضيع الحوسبة على الأسلوب قبل الأمان.
📋التقارير تُنشئ المساءلة — ملفات موثقة تُظهر ما بُني، ما راجعناه، وما أُصلح.
💰حوسبة أكثر = ثقة أكبر — رموز التحقق أرخص بكثير من أخطاء الـ production.
🔗هذا هو القمة — AGENTS.md + Sub-Agents + AI Docs + Skills + Closed Loop = نظام agentic متكامل.
💡 ملخص الـ 4 Levels
L1: أخبر الـ Agent من هو 📄  |  L2: أعطه فريق ومعرفة 👥
L3: وسّع قدراته 🔧  |  L4: اجعله يتحقق من نفسه 🔁
