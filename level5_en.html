📋 Level 5: Bug, Feature & Chore Templates

Class 1, Gear 5 — Different work = Different workflows

"A bug fix that starts with a refactor is a bug fix that will never ship."

1 The Problem: One Agent for everything

Without templates, "fix this bug" and "build this feature" are treated exactly the same by the Agent — and that's wrong.

Work TypePrimary GoalKey Differentiator
🐛 BugFix broken behaviorReproduction first, regression test required
✨ FeatureAdd new capabilitySpec-driven, plan then build
🔧 ChoreMaintenance & cleanupAssess first, verify after execution
📐 PlanDesign the approachRead-only, produces a Spec file

2 Anatomy of a Template

frontmatterdescription + argument-hint → makes the command discoverable in the menu
Variables$1 for the first input, $2 for an optional input
InstructionsStrict constraints on the Agent's behavior
WorkflowNumbered sequential steps — detailed and binding
ReportStructured output format — consistent in every invocation

3 The Four Patterns

🐛 /bug — Reproduction First
1. Reproduce 2. Isolate 3. Fix 4. Verify 5. Regression Test

⚠️ Strict constraint: Minimal fix only — no refactoring surrounding code

✨ /feature — Spec First
1. Spec 2. Plan 3. Build 4. Test 5. Review

🎛️ SCOPE variable: "spec" | "plan" | "full"

🔧 /chore — Assess First
1. Assess 2. Execute 3. Validate 4. Document

⚠️ No change in user-visible behavior unless that is the explicit goal

📐 /plan — Read Only
1. Understand Requirements 2. Read Context 3. Write Spec

🔒 Absolute constraint: Do not write code, never modify a file

📁 Produces: specs/<name>.md

4 Practical Exercise Steps

1
Create the folder
mkdir -p workshop/app/.pi/prompts/
2
Create /bug template

Create bug.md — 5 steps, strict constraint: "Do not pass the reproduction step, and if it fails stop and report".

3
Create /feature template

Create feature.md — 2 variables: $1 description, $2 scope (default: "full"). The Spec is saved in specs/.

4
Create /chore template

Create chore.md — 4 steps. Validate step runs: bun run check + bun run lint + bun test.

5
Create /plan template

Create plan.md — Read-only. Spec includes: Goal, Requirements, Technical Approach, Implementation Steps, Testing, Risks.

🧪 Testing Templates:

/bug "Tasks with status 'done' still appear in the 'To Do' filter"
/feature "Add task labels/tags with filtering" "spec"
/chore "Add input validation to all API endpoints"
/plan "Add task categories with color coding"

5 Key Takeaways

🎯Different work = Different workflow — Bug needs reproduction, Feature needs a Spec, Chore needs assessment.
📏Templates encode best practices — Every invocation follows the same good methodology, regardless of who invokes it.
🎛️Variables provide flexibility — $1 for input, $2 for scope or optional context.
🔍Frontmatter aids discovery — The description appears in the command menu when typing /.
📊Structured reports ensure consistency — Every invocation produces the same report format.
