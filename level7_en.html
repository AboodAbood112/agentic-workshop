🧠 Level 7: Agent Experts with Mental Models

Class 1, Gear 7 — Agents that know your codebase

"The difference between a junior engineer and an expert isn't what they can do — it's what they already know. Give your agents this knowledge."

1 The Starting-From-Scratch Problem

Every time you ask an Agent to work on the backend, it starts from zero:

  • It searches for files → reads them → discovers patterns → understands conventions → then starts the actual work
  • Next session? Same thing — all discovery is lost
⚠️ This is like hiring an expert engineer, wiping their memory after every task, and rehiring them in the morning.

The Solution: Agent Experts — a set of commands powered by a mental model (a structured YAML file) that captures everything the Agent needs to know.

2 The Expert Structure

.pi/prompts/experts/backend/
  expertise.yaml         # The mental model — deep, structured knowledge
  plan.md                # Planning backed by expertise
  question.md            # Answering questions using expertise
  self-improve.md        # Updating the model after every change
  plan_build_improve.md  # Full Chain: Plan → Build → Improve

3 Anatomy of expertise.yaml

📦 overview
description, architecture, runtime, framework
key_patterns: recurring code patterns
🏗️ core_implementation
entry_point: file + lines + responsibilities
routes: file + lines + pattern used
🔌 api_endpoints
create_task:
method: "POST" | path: "/api/tasks"
handler_lines: "38-62"
request_body, error_cases
🗄️ database_schema
columns: type, constraints, format (e.g. UUID v4)
✅ best_practices
Specific, actionable rules

⚠️ Common Mistake

"The backend uses a REST API with a database" — this gives the Agent nothing.
Needed: "POST /api/tasks validates with CreateTaskSchema.safeParse, generates UUID v4, returns 201 with the created task."

6 Practical Exercise Steps

1
Create backend expertise.yaml

Read apps/backend/src/ and document everything precisely: file paths + line numbers + every endpoint + every Zod schema + every database constraint.

2
Create Backend Expert commands

Three files — each starts with "Read expertise.yaml":
question.md → Answer with file/line references
plan.md → Create a Spec in specs/backend-*.md
self-improve.md → Update after git diff + validate YAML

3
Create plan-build-improve chain

Create plan_build_improve.md — links the three automatically via SlashCommand() and Task tool.

4
Create Frontend Expert

Read apps/frontend/src/ — document: data flow, API client, Design System usage, state patterns.

5
Create Design System Expert

Read apps/design-system/src/ — focus on: props, variants, usage patterns for each component.

🧪 Testing:

/experts:backend:question "What validation does the create task endpoint use?"
/experts:backend:plan "Add a PATCH endpoint for partial task updates"
/experts:backend:plan_build_improve "Add a health check endpoint"

7 Key Takeaways

📊expertise.yaml = Structured knowledge, not docs — machine-readable YAML that is specific and actionable.
⚡Mental models eliminate repeated discovery — from O(n) search to O(1) direct lookup.
🔄Self-improvement keeps knowledge accurate — after every change, the Expert updates itself from git diff.
🏠One Expert per Domain — Backend, Frontend, and Design System each have their own patterns.
📈Knowledge accumulates — every plan-build-improve cycle makes the Expert smarter.
