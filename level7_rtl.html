🧠 Level 7: Agent Experts with Mental Models

Class 1, Gear 7 — Agents تعرف قاعدة الكود الخاصة بك

"الفرق بين مهندس مبتدئ وخبير ليس ما يستطيعون فعله — بل ما يعرفونه مسبقاً. أعطِ agents هذه المعرفة."

١ مشكلة البداية من الصفر

كل مرة تطلب من الـ Agent العمل على الـ backend، يبدأ من الصفر:

  • يبحث عن الملفات → يقرأها → يكتشف الأنماط → يفهم الاتفاقيات → ثم يبدأ العمل الحقيقي
  • الجلسة التالية؟ نفس الشيء — كل الاكتشاف ضاع
⚠️ هذا كمن يوظف مهندساً خبيراً، يمحو ذاكرته بعد كل مهمة، ثم يعيد توظيفه في الصباح.

الحل: Agent Experts — مجموعة أوامر مدعومة بـ mental model (ملف YAML منظم) يلتقط كل ما يحتاج الـ Agent معرفته.

٢ بنية الـ Expert

.pi/prompts/experts/backend/
  expertise.yaml         # النموذج الذهني — معرفة عميقة ومنظمة
  plan.md                # تخطيط مدعوم بالخبرة
  question.md            # إجابة أسئلة بالرجوع للخبرة
  self-improve.md        # تحديث النموذج بعد كل تغيير
  plan_build_improve.md  # Chain كاملة: خطط → ابنِ → تحسّن

٣ تشريح ملف expertise.yaml

📦 overview
description، architecture، runtime، framework
key_patterns: أنماط الكود المتكررة
🏗️ core_implementation
entry_point: ملف + أسطر + مسؤوليات
routes: ملف + أسطر + النمط المستخدم
🔌 api_endpoints
create_task:
method: "POST" | path: "/api/tasks"
handler_lines: "38-62"
request_body، error_cases
🗄️ database_schema
columns: النوع، القيود، الشكل (مثلاً UUID v4)
✅ best_practices
قواعد محددة وقابلة للتطبيق مباشرة

⚠️ خطأ شائع

"الـ backend يستخدم REST API مع قاعدة بيانات" — هذا لا يفيد الـ Agent شيئاً.
المطلوب: "POST /api/tasks يتحقق بـ CreateTaskSchema.safeParse، يولّد UUID v4، يرجع 201 مع الـ task المنشأ".

٦ خطوات التمرين العملي

١
أنشئ backend expertise.yaml

اقرأ apps/backend/src/ ووثّق كل شيء بدقة: مسارات الملفات + أرقام الأسطر + كل endpoint + كل Zod schema + كل constraint في قاعدة البيانات.

٢
أنشئ أوامر الـ Backend Expert

ثلاثة ملفات — كل واحد يبدأ بـ "اقرأ expertise.yaml":
question.md → أجب مع مراجع ملفات/أسطر
plan.md → أنشئ Spec في specs/backend-*.md
self-improve.md → حدّث بعد git diff + تحقق من YAML

٣
أنشئ plan-build-improve chain

أنشئ plan_build_improve.md — يربط الثلاثة تلقائياً عبر SlashCommand() و Task tool.

٤
أنشئ Frontend Expert

اقرأ apps/frontend/src/ — وثّق: تدفق البيانات، API client، استخدام Design System، أنماط الـ state.

٥
أنشئ Design System Expert

اقرأ apps/design-system/src/ — ركّز على: الـ props، الـ variants، أنماط الاستخدام لكل component.

🧪 اختبار:

/experts:backend:question "What validation does the create task endpoint use?"
/experts:backend:plan "Add a PATCH endpoint for partial task updates"
/experts:backend:plan_build_improve "Add a health check endpoint"

٧ الدروس المستفادة

📊expertise.yaml = معرفة منظمة لا وثائق — YAML قابل للقراءة آلياً ومحدد وقابل للتطبيق مباشرة.
⚡النماذج الذهنية تُلغي الاكتشاف المتكرر — من O(n) بحث إلى O(1) بحث مباشر.
🔄التحسين الذاتي يُبقي المعرفة دقيقة — بعد كل تغيير، الـ Expert يحدّث نفسه من git diff.
🏠خبير واحد لكل Domain — Backend وFrontend وDesign System لكل منها أنماطه الخاصة.
📈المعرفة تتراكم — كل دورة plan-build-improve تجعل الـ Expert أذكى.
