🌐 Level 8: Out-Loop Systems & Webhooks

Class 2, Gear 1 — The Agent working without you

"The moment an Agent responds to events instead of commands, you've crossed the line from a tool to a system."

1 The Shift: In-Loop → Out-of-Loop

❌ In-Loop (Levels 1-7)
You → Agent → Result → You review

Requires your presence — doesn't work at 3 AM

✅ Out-of-Loop
Event → Webhook → Agent → Notification

Runs automatically — you are the designer, not the operator

🔥 Examples of Out-of-Loop:

  • A PR is opened on GitHub → Agent reviews it automatically
  • A Slack message arrives → Agent starts a build workflow
  • A monitoring alert fires → Agent investigates and reports

2 The PITER Framework

Every out-of-loop workflow follows the same 5 steps:

P
Prompt
Define the task for the Agent
I
Input
Gather the necessary context
T
Transform
Prepare the data for the Agent
E
Execute
Run the PI command
R
Report
Send the result to the source
StepWhat HappensExample

TransformData cleaningConvert diff to markdown, extract modified files
ExecuteRun PIpi -p "/review <context>"
ReportResult deliveryPost a review comment on the PR

⚠️ Common Mistake

Skipping the Transform step and sending a raw payload directly to the Agent. Raw data is full of noise — Transform is where you clean the context.

💡 Separation of Concerns: The Webhook Server is generic and stable. Handler Scripts are specific and evolve. Add new workflows by adding handlers, not by modifying the Server.

4 Practical Exercise Steps

1
Create Webhook Server

Build an HTTP server in Python that receives POST requests on /webhook, parses JSON, and routes to the correct handler based on event type.

curl -X POST http://localhost:8080/webhook \
  -d '{"event": "bug_report", "description": "Tasks not saving"}'
2
Add GitHub Integration

Handle two events:

  • pull_request (opened) → code review workflow
  • issues (bug label) → bug investigation workflow

Note: GitHub sends the event type in the header: X-GitHub-Event

3
Add Slack Integration

Handle 3 slash commands:

  • /taskflow-build <desc> → build workflow
  • /taskflow-review → review workflow
  • /taskflow-status → system status

Slack sends form-encoded data, not JSON — use parse_qs(). Respond within 3 seconds.

4
Implement Full PITER per Handler

Execute Step:

subprocess.run(["pi", "-p", prompt], capture_output=True)

Transform Step: Build clean markdown with only the required context.

5
Add Error Handling & Logging

Wrap every handler in a try/except block. Log every PITER step with timestamps. Send a failure report to the source even on error — silent failures are the worst kind.

5 Key Takeaways

🎯Out-of-loop is the goal — An Agent that needs your presence for every task is a tool. An Agent that responds to events automatically is a system.
🏗️PITER provides structure — 5 sequential steps for every out-of-loop workflow.
✨Transform is the value generator — Raw data is noisy. Transform delivers exactly what the Agent needs.
🔌Separation of concerns — Generic server + Specific handlers = add workflows without architectural changes.
🚨Always report errors — The Report step is vital, especially for errors. Silently failing automated systems are the worst.
💡 Summary of the 8 Levels
L1-L7: You are in the loop — you write a command, the Agent executes
L8: You are out of the loop — events trigger the Agent automatically 🌐
