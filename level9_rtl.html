⚙️ Level 9: AI Developer Workflows (ADWs)

Class 2, Gear 2 — كود حتمي يُنسّق Agents غير حتمية

"كود حتمي يُنسّق Agents غير حتمية. أعلى نقطة نفوذ في البرمجة الـ Agentic."

١ المعمارية الهجينة

Pure Code
✅ متوقع وسريع
✅ قابل للاختبار
❌ لا يتعامل مع المشاكل الجديدة
❌ هش أمام التغيير
Pure Agent
✅ إبداعي وتكيفي
✅ يتعامل مع الغموض
❌ يفقد السياق
❌ يتخطى خطوات
🏆 ADW (Hybrid)
✅ بنية متوقعة
✅ تنفيذ إبداعي
✅ لا يمكن تخطي خطوة
✅ audit trail كامل

نمط ADW:

Python (حتمي) → الهيكل العظمي — يضمن ترتيب الخطوات
Model (غير حتمي) → العقل — يُفكر ويكتب الكود

الهيكل العظمي يضمن عدم تخطي العقل لأي خطوة أو تغيير الترتيب

⚠️ خطأ شائع

محاولة تنفيذ workflows متعددة الخطوات عبر prompt engineering فقط. "أولاً خطّط، ثم ابنِ، ثم راجع" في prompt واحد هو مجرد اقتراح — الـ Agent يتجاهله.

٢ دورة حياة كل خطوة

log_step_start(step_name) # دائماً يحدث
│
├── Construct prompt for this step
├── subprocess.run(["pi", "-p", command])
├── Capture stdout/stderr + exit code
│
log_step_end(step_name, result) # دائماً يحدث

يعطيك ثلاثة أشياء لا يملكها Pure Agent:

  • Observability كاملة — سجلات مؤرخة لبداية ونهاية كل خطوة
  • حدود واضحة — كل خطوة استدعاء model جديد، لا تسرب سياق
  • قدرة الإعادة — يمكن إعادة خطوة واحدة فاشلة فقط

٣ تركيبات الـ Workflows

2 خطوات: Plan→ Build للميزات السريعة
3 خطوات: Plan→ Build→ Review مع ضمان الجودة
4 خطوات: Plan→ Build→ Review→ Fix (إذا FAIL) أتمتة كاملة مع إصلاح ذاتي

🔀 نمط Fix الشرطي:

review_result = run_step("review")
if review_result.verdict == "FAIL":
    run_step("fix")
    run_step("review")  # إعادة المراجعة بعد الإصلاح
    # حد أقصى لعدد المحاولات — عادةً 2-3

Python يتخذ القرار الشرطي. الـ Model ينفّذ فقط داخل الحدود.

⚠️ خطأ شائع

حلقة review-fix-review بدون حد أقصى. ضع دائماً حداً للمحاولات — إذا لم يُصلح الـ Agent في 3 محاولات، يحتاج تدخلاً بشرياً.

٤ هيكل الملفات

adws/
  adw_modules/
    adw_runner.py          # المحرك — ينفّذ الخطوات مع logging
    adw_logger.py          # كتابة JSON logs مؤرخة
  adw_workflows/
    adw_plan_build.py           # 2 خطوات
    adw_plan_build_review.py    # 3 خطوات
    adw_plan_build_review_fix.py # 4 خطوات + شرط
  adw_triggers/
    run_adw.py             # CLI trigger
    run_adw_background.py  # تشغيل في الخلفية

٥ خطوات التمرين العملي

١
أنشئ Logger Module

أنشئ adw_logger.py — يكتب JSON logs لكل خطوة (timestamp + اسم الخطوة + نوع الحدث + النتيجة). سطر JSON واحد لكل حدث.

٢
أنشئ Runner Module

أنشئ adw_runner.py — يأخذ قائمة خطوات، ينفّذ كل خطوة بـ subprocess.run(["pi", "-p", command])، يُسجّل قبل وبعد.

٣
أنشئ 2-Step Workflow

أنشئ adw_plan_build.py:

STEPS = [
    {"name": "plan", "command": "/plan", ...},
    {"name": "build", "command": "/build", ...}
]
runner.run(steps=STEPS, prompt=user_prompt)
٤
أضف Review → 3 خطوات

أنشئ adw_plan_build_review.py — نفس الـ 2-step + إضافة /review كخطوة ثالثة.

٥
أضف Fix الشرطي → 4 خطوات

أنشئ adw_plan_build_review_fix.py — إذا Review = FAIL، شغّل Fix ثم أعد Review. بحد أقصى للمحاولات.

🧪 اختبار:

python adws/adw_triggers/run_adw.py \
    --workflow plan_build \
    --prompt "Add task filtering by status" \
    --working-dir /path/to/workshop/app

٦ الدروس المستفادة

🤝الحتمي يُغلّف غير الحتمي — Python يتحكم بالتدفق، الـ Model يتحكم بالتفكير. لا شيء يعمل مكان الآخر جيداً.
🚧الخطوات حدود لا اقتراحات — كل خطوة استدعاء process منفصل. الـ Agent يستحيل عليه تخطيها.
👁️Observability مدمجة — step lifecycle logging يعني أنك تعرف دائماً ما حدث ومتى.
🧩التركيب = التوسيع — Workflows جديدة هي مجرد قوائم خطوات جديدة. البنية التحتية معادة الاستخدام.
🐍المنطق الشرطي في Python — هل تُشغّل Fix؟ Python يقرر. هل التنفيذ إبداعي؟ الـ Model يقرر.
